ONNX Conversion and Optimization

The Universal Model Format
From PyTorch to Everywhere

Understanding ONNX as a bridge between frameworks and hardware

You've trained your model in PyTorch, but now you need to deploy it to different hardware platforms - some prefer TensorRT, others like Qualcomm's Hexagon NN, and maybe you want to try OpenVINO too. Maintaining separate export pipelines for each target is tedious. Enter ONNX - the Open Neural Network Exchange format that acts as a universal intermediary.

What is ONNX?

ONNX (Open Neural Network Exchange) is an open format built to represent machine learning models. ONNX defines a common set of operators - the building blocks of machine learning and deep learning models - and a common file format to enable AI developers to use models with a variety of frameworks, tools, runtimes, and compilers.

Why Use ONNX?

ONNX provides several key benefits for computer vision deployment:

  • Framework Interoperability: Train in PyTorch, deploy with TensorRT, OpenVINO, or Qualcomm SNPE
  • Hardware Optimization: Each platform can optimize the ONNX model for their specific accelerators
  • Versioning and Compatibility: Clear specification helps avoid framework version lock-in
  • Community and Ecosystem: Growing support from major tech companies and hardware vendors
  • Extensibility: Custom operators for specialized hardware or novel layer types

The ONNX Conversion Process

Converting a PyTorch model to ONNX involves these steps:

  1. Model Preparation: Ensure model is in evaluation mode
  2. Dummy Input: Create sample input tensor with correct shape
  3. Export: Use torch.onnx.convert to export the model
  4. Verification: Check that the ONNX model produces same outputs as PyTorch model
  5. Optimization: Use ONNX Runtime tools to optimize the model

Basic ONNX Export in PyTorch

import torch
import torch.onnx

# Set model to evaluation mode
model.eval()

# Create dummy input (batch_size, channels, height, width)
dummy_input = torch.randn(1, 3, 224, 224, requires_grad=True)

# Export the model
torch.onnx.export(
    model,                     # model being run
    dummy_input,               # model input (or tuple for multiple inputs)
    "model.onnx",              # where to save the model
    export_params=True,        # store the trained parameter weights
    opset_version=11,          # the ONNX version to export the model to
    do_constant_folding=True,  # whether to execute constant folding
    input_names=['input'],     # the model's input names
    output_names=['output'],   # the model's output names
    dynamic_axes={             # variable length axes
        'input' : {0 : 'batch_size'},    # variable length axes
        'output' : {0 : 'batch_size'}
    })

ONNX Runtime and Optimization

After conversion, you can optimize and run the model with ONNX Runtime:

  • ONNX Runtime: Cross-platform inference engine with performance optimizations
  • Graph Optimization: Fuse operations, eliminate redundant nodes, optimize memory usage
  • Execution Providers: Hardware-specific backends (CUDA, TensorRT, QNN, etc.)
  • Quantization: Built-in quantization tools for INT8/FP16 conversion
  • Profiling: Tools to identify performance bottlenecks

Platform-Specific ONNX Workflows

How different platforms utilize ONNX:

NVIDIA Orin NX with TensorRT

  • ONNX → TensorRT parser → optimized engine
  • Leverage TensorRT's layer fusion, kernel auto-tuning, and precision calibration
  • Supports FP16, INT8, and FP32 precision modes
  • Use trtexec for benchmarking and validation

Qualcomm QCS8550 with QNN

  • ONNX → Qualcomm Neural Processing SDK (QNN)
  • Hexagon NNP (Neural Network Processor) and HTA (Tensor Accelerator) support
  • Optimize for Hexagon DSP vector units and scalar processors
  • Use SNPE (Snapdragon Neural Processing Engine) for deployment

Cross-Platform with ONNX Runtime

  • Single runtime that works across platforms
  • Choose execution provider based on available hardware
  • Good for prototyping or when hardware-specific SDKs aren't available
  • Supports CPU, CUDA, TensorRT, OpenVINO, and QNN execution providers

Common Issues and Solutions

When working with ONNX, watch out for these challenges:

  1. Operator Support: Some PyTorch ops may not have ONNX equivalents - use custom operators or approximate
  2. Dynamic Shapes: Models with variable input sizes need special handling for static shapes
  3. Numerical Differences: Small numerical differences between frameworks - usually acceptable
  4. Training vs Inference: Ensure dropout/batch norm behave correctly in inference mode
  5. Custom Layers: Novel architectures may require exporting custom ONNX operators
