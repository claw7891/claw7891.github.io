Real-time Inference and Tracking

Making Your Model See in Real Time
From Single Images to Video Streams

Techniques for achieving low-latency, high-frame-rate inference

You've deployed your model to the Orin NX or QCS8550, but now you notice it's only running at 5 FPS when you need 30 FPS for smooth tracking. In this lesson, we'll learn how to optimize your inference pipeline for real-time performance and implement object tracking to maintain identity across frames.

Understanding Latency

Real-time inference requires minimizing latency at every stage:

  • Image Capture: Camera exposure time and readout speed
  • Preprocessing: Resizing, normalization, color conversion
  • Inference: Model computation time on accelerator
  • Postprocessing: NMS, decoding predictions, filtering
  • Display/Output: Rendering results or sending to next stage

Optimizing the Inference Pipeline

Here are strategies to reduce latency:

Batch Size = 1

For real-time, process one frame at a time to avoid waiting for batch accumulation.

Asynchronous Processing

Overlap capture, preprocessing, inference, and postprocessing using multiple threads or pipelines.

Hardware-Accelerated Preprocessing

Use GPU or dedicated ISP for resizing, color conversion, and normalization.

Model Input Shape Optimization

Choose input resolution that balances accuracy and speed (e.g., 416x416 vs 608x608 for YOLO).

Object Tracking Fundamentals

Detection alone isn't enough for many applications - you need to maintain object identity across frames:

  • Why Track? To count objects, analyze behavior, reduce false positives, and enable smoother interaction
  • Tracking by Detection: Simple approach - detect in each frame and associate detections using IOU or distance
  • Kalman Filters: Predict object motion based on previous state to handle temporary occlusions
  • Deep SORT: Combines deep appearance features with Kalman filtering for robust tracking
  • ByteTrack: Associates low-confidence detections to improve tracking of small/fast objects

Embedded-Specific Tracking Considerations

On embedded systems like Orin NX and QCS8550:

  1. Memory Bandwidth: Tracking algorithms that require large feature maps may strain memory bandwidth
  2. Compute Budget: Appearance-based tracking (like Deep SORT) adds significant compute overhead
  3. Latency Trade-offs: More sophisticated tracking increases latency - choose based on your FPS requirements
  4. Hardware Acceleration: Some tracking components can be offloaded to GPU, NPU, or DSP

Practical Implementation Tips

  • Start simple: use IOU-based tracking before adding complex appearance models
  • Measure end-to-end latency, not just model inference time
  • Consider region-of-interest (ROI) processing to reduce computation in irrelevant image areas
  • Use hardware-accelerated video decoding/encoding if working with compressed video streams
  • Implement fallback mechanisms for tracking failures (e.g., redetection after loss)
